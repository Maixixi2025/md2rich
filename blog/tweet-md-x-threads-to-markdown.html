tweet.md + md2rich：30 秒把任何 X 线程变成博客文章

2026 年 6 月上线了一个极简工具 tweet.md（tweet.md）：输入任何 X（Twitter）线程 URL，一键提取所有推文并整理成 Markdown。搭配 md2rich 的富文本渲染 + 复制功能，你可以在 30 秒内把别人的 X 线程变成自己的 LinkedIn 文章、博客草稿或 X Articles 长文。

这件事值得一篇独立文章，因为它是今年出现的最干净的 X → Markdown → 富文本 全链路工具链。本文实测 tweet.md 的核心能力，展示它与 md2rich 的配合方式，并对比其他 3 种 X 线程转 Markdown 方案。

1. tweet.md 是什么

tweet.md 是一个极简的在线工具。打开 tweet.md，页面就只有一行输入框 + 一个按钮。输入 X 线程 URL，点击"Fetch"，几秒钟后，页面下方整条线程就以 Markdown 格式呈现。

不需要登录。不需要 API key。不需要注册。它在 2026 年 6 月 5 日上线，在 Show HN 迅速获得 100+ 分，理由是"干净利落，正好填上 X 生态系统里 Markdown 用户最需要的一个缺口"。

输出样例

用 tweet.md 提取一条 5 条推文的线程，输出长这样：

# Why we moved from Redis to Dragonfly\n\nAfter 3 years on Redis 7.0, our team migrated to Dragonfly.\nHere is what we learned.\n\n---\n\n**1. Memory efficiency**\n\nRedis stores everything in a single-threaded event loop.\nDragonfly uses multi-threaded shared-nothing architecture.\nFor our 64GB workload, RSS dropped from 51GB to 29GB.\n\n---\n\n**2. Connection handling**\n\nAt 10k connections, Redis showed 120ms p99 latency.\nDragonfly at 10k: 18ms p99.\n\nThe `maxclients` cap on Redis became a bottleneck\nwe had to configure, Dragonfly handles it natively.\n\n---\n\n**3. Migration path**\n\nDragonfly speaks the Redis protocol.\nWe pointed our app at the Dragonfly port, ran a snapshot\nimport, and the app never noticed.\n\n---\n\n**4. Downsides**\n\n- Dragonfly's Lua scripting is not a 1:1 replacement\n- Snapshot export format is different\n- The community is smaller, fewer plugins\n\n---\n\n**5. Verdict**\n\nCache-heavy workloads: Dragonfly.\nPersistent + pub/sub: stick with Redis.\n\nvia @username

每次推文用 --- 分隔，每条的序号自动加上粗体标题。作者的 @ 出现在底部。整个输出就是干净可编辑的 Markdown——没有冗余 HTML、没有推特自带的 t.co 追踪链接（链接已是真实 URL）、没有时间戳噪音。

2. 完整工作流：tweet.md → md2rich → 发布

这是从 X 线程到博客文章的最短路径：

  1. 在 X 上找到一条高质量的线程
  2. 复制线程的 URL（例如 https://x.com/username/status/123456789）
  3. 打开 tweet.md，粘贴 URL，点击"Fetch"
  4. 复制输出的 Markdown 全文
  5. 打开 md2rich.com，把 Markdown 粘贴到左侧编辑器
  6. 右侧实时预览渲染效果
  7. 点击"Copy as Rich Text"，切换到 LinkedIn / X Articles / Medium
  8. 按 Cmd+V（或 Ctrl+V）直接粘贴

全程不需要启动任何桌面软件、不需要登录任何服务、没有任何数据离开你的浏览器（md2rich 无后端）或只经过 tweet.md 一次（它也不保存任何内容）。

3. 为什么要这条链路

X 上的技术线程质量很高。很多开发者习惯在 X 上"写书"——用一条接一条的推文发布深度长文。但 X 线程有几个致命缺点：

  • 不可搜索：Google 很难完整索引一条多推文线程
  • 不可引用：别人转发时只能抄文字，没有结构化引用
  • 不可归档：账号删了线程就没了
  • 缺乏 SEO：X 线程几乎不会出现在搜索引擎结果中

把 X 线程转成博客文章发表到 LinkedIn、Medium 或你自己的站点，能突破这些限制。这就是 tweet.md + md2rich 解决的问题。

4. 三种替代方案横向对比

除了 tweet.md，还有几种方式可以把 X 线程转成 Markdown：

方案 A：tweet.md（推荐）

优势：零配置、无需登录、输出干净、实时。唯一输入是 X 线程 URL，唯一输出是 Markdown。没有多余的步骤。

方案 B：nitter.net（开源替代前端）

Nitter 是 X 的开源替代前端。打开 https://nitter.net/username/status/123456789，可以看到线程的纯文本版本。但 nitter 输出的是 HTML，不是 Markdown。要转成 Markdown 需要额外用 Chrome 扩展（如 MarkDownload）或手动复制粘贴。

问题：nitter 实例经常被 X 封禁、镜像站频繁换域名、输出格式在不同实例间不一致。2026 年可用的稳定实例越来越少。

方案 C：ThreadGrab 浏览器扩展

ThreadGrab 是一个浏览器扩展，在 X 页面点击按钮即可抓取当前线程，导出 Markdown、CSV 或纯文本。优势是可以批量操作——一次性抓取某作者的全部线程。

缺点：需要安装浏览器扩展、只能从浏览器里交互操作、没有公开 API 供脚本调用。对于单条线程的快速提取，不如 tweet.md 方便。

方案 D：自建 Python 脚本

用 snscrape 或 twikit 库可以编写 Python 脚本抓取 X 线程并格式化输出 Markdown。优势是完全自动化、可定制。缺点是需要维护 API 兼容性（X API 变化频繁）、需要处理登录/认证、不是每个人都会 Python。

6. 关于数据的隐私问题

如果你转发的是别人的公开线程（技术分享、行业分析），隐私不是问题。但如果你在整理自己尚未发布的草稿线程，或者涉及内部讨论的线程，就需要考虑数据流转路径：

  • tweet.md：你输入的 X URL 会发到它的服务端去抓取。根据开发者的描述，服务端不做持久化，但内容在传输过程中会被解析一次
  • md2rich：100% 纯前端，数据从不离开你的浏览器。粘贴 Markdown 后，内容只存在于当前页面的内存中，关闭标签页即消失
  • 最终发布平台（LinkedIn/X Articles/Medium）：数据会上传到平台服务器。这是发布流程中不可避免的

简而言之：tweet.md 用于一次性提取，md2rich 在本地渲染，你只在最终发布时才把数据上传到目标平台。这一链路比"用 ChatGPT 改写线程"（内容发给 OpenAI）或"用第三方发布工具一键同步"（内容长期存储在第三方服务器）更安全。

FAQ

tweet.md 支持中文线程吗？

支持。X 线程的语言不影响提取。中文、英文、日文主题的线程都能正常提取为 Markdown。emoji 也能保留。

提取的 Markdown 能直接贴到 md2rich 里吗？

可以。tweet.md 输出的就是标准 Markdown，md2rich 完全兼容。你粘贴后右侧立刻渲染，如果对标题层级或分隔线效果不满意，可以在 md2rich 里编辑后再复制富文本。

tweet.md 会不会有 X API 限制问题？

tweet.md 似乎使用了网页抓取而非 X API，暂时没有遇到速率限制。但如果 X 增加了反爬措施，tweet.md 的稳定性可能会受影响——这也是为什么建议把 tweet.md + md2rich 定位为"日常顺手用"而不是"生产级自动化 pipeline"的核心工具。
