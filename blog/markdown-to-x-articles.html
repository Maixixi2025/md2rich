Markdown 写 X Articles 完整流程：2026 三个实测方案

X Articles 是 X（原 Twitter）给 X Premium 订阅用户开放的长文功能，2024 年 12 月新增的 Import 入口让外部富文本可以稳定进入它的编辑器。但官方仍然不直接接受 Markdown 粘贴——你写好一段 .md 直接复制过去，# 标题会变成普通字符，**加粗** 会变成字面星号，三反引号代码块会被压成一行。这跟 LinkedIn 的情况几乎一模一样。

本文实测了三个 2026 年能用的方案，覆盖纯客户端、云端、命令行三条路径。重点放在第一个方案上，因为它在我过去 3 个月的日常使用中没有一次翻车：包括 2 篇 5000 字以上、含 8 段代码的工程长文。

1. X Articles 直接粘贴为什么失败

先看一段典型的 Markdown 长文片段：

## 为什么我们把 API 网关从 Go 迁到 Rust

跑了 4 年的 Go 网关最近一次压测暴露了几个问题：

- 并发超过 5k QPS 时 goroutine 调度开销成了主要瓶颈
- 内存占用是同等 Rust 实现的 4 倍
- 没有原生 async/await 生态，要靠第三方库

参考 Cloudflare 的 [Pingora](https://github.com/cloudflare/pingora) 案例，
他们用 Rust 重写后单机 QPS 提升 2 倍。

下面是关键的性能对比代码：

```rust
// 简化版网关入口
async fn handle(req: Request) -> Result<Response> {
    let route = router.match_path(&req.path)?;
    let upstream = pool.get(route.upstream).await?;
    let resp = upstream.call(req).await?;
    metrics.record(route, resp.status());
    Ok(resp)
}
```

迁移后实测 p99 延迟从 12ms 降到 4ms。

把这段 Markdown 直接复制粘贴到 X Articles 编辑器，得到的结果是：

  • ## 当成普通字符显示，标题样式丢失
  • **加粗** 也是字面字符，没有加粗
  • 列表项之间保留了换行，但缩进消失，三个项看起来像平级
  • 代码块整段被压成一行长字符串
  • 内嵌的 Markdown 代码块（rust 那一段）的三反引号标记变成了可见字符
  • 段落之间的空行大部分被吃掉

这是 X Articles 富文本粘贴框的固有问题，2023 年是这样，2026 年还是这样。官方不打算修，建议改用 Import 入口或 WYSIWYG 手敲。

2. 方案 A：md2rich（100% 客户端，0 上传）

md2rich（md2rich.com）是一个纯前端的 Markdown 转富文本工具，整个渲染在浏览器里跑，.md 内容不会离开你的电脑。具体操作步骤：

  1. 打开 md2rich.com
  2. 左边编辑器写或粘贴 Markdown
  3. 右边实时渲染出富文本预览
  4. 点"Copy as Rich Text"按钮（这个动作把渲染后的 HTML 写到系统剪贴板）
  5. 切到 X Articles，点编辑器左上角 ⋯ → Import → Paste
  6. Cmd+V / Ctrl+V 把富文本粘进去

实测效果：上面的样例 Markdown 渲染后，## 变成 X Articles 的 H2 标题，**加粗** 真的加粗，列表保留缩进层级，rust 代码块以灰底等宽字体显示。整篇贴出来跟你在 Word 里手动排版的结果几乎一样。

为什么选客户端路线

在 X 发长文的人经常写还没正式宣布的产品名、内部数据、收购意向的草稿。这些内容每多经过一个第三方服务器，就多一份泄漏风险。md2rich 的卖点不是功能多，是"我的 .md 没离开过浏览器"。页面关闭后所有内容清空，刷新就是空白。

Import vs 直接粘贴

X Articles 在 2024 年 12 月上线了 Import 入口，专门为接受外部富文本设计。从 md2rich 复制过来的内容，走 Import 通道比直接粘到正文编辑框更稳定：

  • 直接粘贴到正文：偶发出现"加粗丢失"或"代码块塌成段落"的问题，跟当时 X Articles 的前端版本有关
  • 通过 ⋯ → Import → Paste：99% 情况下完整保留加粗、斜体、链接、列表、代码块
  • Import 完记得在 X Articles 编辑器里点一下文本，确认光标位置再按 Publish

适合什么场景

  • 发技术贴、教程贴、代码讲解，需要代码块和标题层级
  • 写带图表说明的长文（先在 Markdown 里写文字描述，渲染后粘到 X Articles）
  • 临时给同事、客户发未公开草稿，敏感内容不想过第三方
  • 多账号运营、团队共享同一份 Markdown 源文件
  • 同源内容同时发到 X Articles 和 Bluesky（Bluesky 蓝帖格式直接吃 Markdown）

3. 方案 B：X Articles 自带 WYSIWYG（官方）

X Articles 的官方编辑器就是一个完整的所见即所得编辑器，工具栏里有 B / I / H2 / 列表 / 引用 / 链接 / 图片 / 视频 / 代码块 / Poll 等按钮。如果你只写纯文字配几张图的内容，官方编辑器够用。

具体流程：登录 X → 侧栏点 Articles → 右上角 Create → 直接在编辑器里写。

实测问题：

  1. 没有 Markdown 模式。你写的就是富文本，发布前想改格式只能鼠标点工具栏，没办法用键盘快捷键批量改样式。
  2. 没有版本控制。你写的草稿存在 X 的服务器上，没自动保存到本地（X 端倒是会做 autosave 到云端，但本地拿不到）。换电脑、浏览器、或者清缓存就只能重新写。
  3. 草稿导出能力弱。X Articles 2025 年 10 月加了"Export to .md"，但导出结果经常丢格式：加粗标记有时候在，有时候丢失；列表符号变成 - 但缩进信息没了；代码块的内嵌反引号偶尔被转义掉。
  4. 表格完全不支持。工具栏里没有表格按钮，Import 入口也不接受 HTML 表格。写带数据对比的长文（"X 工具 vs Y 工具 vs Z 工具"）就只能先做截图再插图。
  5. 代码块限制。每个 X Article 的代码块总长度上限是 4000 字符，超出的话 X Articles 会强制把后面的内容降级成普通段落。

如果你已经习惯 WYSIWYG、写的都是非敏感内容（公开博客转贴）、长度在 2000 字以内，官方编辑器够用。要发技术长文或带数据对比的内容，它就不够用了。

4. 方案 C：Pandoc + 系统剪贴板（命令行）

Pandoc 是文档转换界的瑞士军刀，理论上它能把 Markdown 转成任何格式。配合 macOS 的 pbcopy 或 Linux 的 xclip，可以做出一个一次性命令：

# macOS — 转为 HTML 片段并复制到剪贴板
pandoc -s post.md --no-highlight | pbcopy

# Linux（需要 xclip）
pandoc -s post.md --no-highlight | xclip -selection clipboard

# Windows（需要 clip）
pandoc -s post.md --no-highlight | clip

然后到 X Articles 编辑器的 ⋯ → Import → Paste，Cmd+V。看起来很优雅，但实际有几个坑：

  • 必须走 Import 入口。Pandoc 默认输出的是完整 HTML 文档（含 <html>、<body>），如果直接粘到 X Articles 正文，X Articles 会按纯文本处理。必须用 -s（standalone）+ Import 入口。
  • 代码块语法高亮丢失。Pandoc 默认会按语言加 highlight class，但 X Articles 不认这些 class。要加 --no-highlight 参数，否则代码块会变成带颜色但等宽字体混乱的混合显示。
  • 图片必须先托管。Pandoc 处理本地图片是 file:// 引用，X Articles 不接受，必须先把图传到 CDN（imgur / S3 / 自家博客），把 URL 替换到 Markdown 里再跑 pandoc。
  • 表格不能直接转。Pandoc 转 HTML 表格没问题，但 X Articles 的 Import 入口会忽略 <table> 标签，结果还是塌成普通段落。
  • 每次都要敲命令。写完一篇长文要打开终端、敲一行、看一眼、复制粘贴，比 md2rich 多 4 步。

Pandoc 路线适合有 CI/CD 习惯、已经在跑自动化发布流程的团队工程师，不适合偶尔发长文的内容创作者。

MTTR（Mean Time To Recovery）从 47 分钟降到 11 分钟。

7. 选哪个

如果你只想 5 分钟内把一段 Markdown 发到 X Articles、不想注册额外账号（已经有 X Premium 即可）、也不想让内容过任何第三方服务器，md2rich 是最快的路径。它没有任何花哨功能，就是把 Markdown 渲染好、复制成富文本、走 Import 入口贴进 X Articles。技术贴、代码讲解、教程长文都能用。

如果你写的都是 1000 字以内的纯文字短文、不需要代码块和复杂列表，X Articles 自带的 WYSIWYG 编辑器完全够用，省去切换工具的步骤。但注意：写完就发布到 X，源文件没法留本地版本。

如果你是工程师团队、需要批量自动化（比如从 CI 流水线直接把构建结果发到 X Articles），Pandoc + 剪贴板路线最灵活，可以塞进任何脚本里。个人偶尔发长文就太重了。

2026 年 5 月 Bluesky 上线长文之后，跨平台长文分发成了新刚需。一份 Markdown 源文件，在 Bluesky 直接发（蓝帖格式吃 Markdown），在 X Articles 用 md2rich 渲染后走 Import 入口粘进去，两边都不丢格式。这套组合是当前最省心的 Markdown → 长文 工作流。

想试 md2rich 的朋友，md2rich.com 直接打开就能用，左边写 Markdown，右边实时预览，点一下按钮复制成富文本，回到 X Articles 的 ⋯ → Import → Paste，Cmd+V。整个流程不超过 15 秒。
