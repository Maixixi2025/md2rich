Markdown 写 LinkedIn 富文本：2026 三个实测靠谱方案

2026 年还在用 Word 写 LinkedIn 帖子的人，大概率经历过这种场景：辛辛苦苦排好三级标题、加粗关键词、贴上代码块，复制到 LinkedIn 编辑器一看，全变成纯文本，# 符号留着，**加粗标记** 留着，三反引号代码块被压成一行。LinkedIn 的富文本粘贴框对 Markdown 几乎不友好，必须先在外部把 .md 渲染成富文本再贴进来。

本文实测了三个 2026 年可用的方案，覆盖纯客户端、云端、命令行三条路径。重点放在第一个方案上，因为它在我 6 个月的日常使用中没有一次翻车。

1. 直接粘贴为什么失败

先看一段典型的 Markdown 帖子：

## 为什么我们的 API 慢了 3 倍

上周我们在生产环境发现一个奇怪的现象：
同一个 endpoint，p99 延迟从 180ms 涨到了 540ms。

排查思路：

1. 先看监控曲线，确认是单 AZ 还是全局
2. 检查最近的发布记录，回滚到上一个稳定版本
3. 在 staging 重现问题，抓 pprof

**结论**：上游 Redis 实例 CPU 打满了，触发了连接池排队。

把这段 Markdown 直接复制粘贴到 LinkedIn 编辑器，得到的结果是：

  • ## 当成普通字符显示，标题样式丢失
  • **结论** 也是字面字符，没有加粗
  • 列表自动保留了换行，但缩进消失，三个项看起来差不多
  • 代码块整段被压成一行长字符串
  • 段落之间的空行大部分被吃掉

这是 LinkedIn 富文本粘贴的固有问题，2023 年是这样，2026 年还是这样。官方不打算修，写过工单的作者都被告知去用他们的 WYSIWYG 编辑器手敲。

2. 方案 A：md2rich（100% 客户端，0 上传）

md2rich（md2rich.com）是一个纯前端的 Markdown 转富文本工具，整个渲染在浏览器里跑，.md 内容不会离开你的电脑。具体操作步骤：

  1. 打开 md2rich.com
  2. 左边编辑器写或粘贴 Markdown
  3. 右边实时渲染出富文本预览
  4. 点"Copy as Rich Text"按钮（这个动作把渲染后的 HTML 写到系统剪贴板）
  5. 切到 LinkedIn 编辑器，Cmd+V / Ctrl+V 粘贴

实测效果：上面的样例 Markdown 渲染后，## 变成 LinkedIn 的一级标题，**结论** 真的加粗，列表保留缩进层级，代码块以灰底等宽字体显示。整篇贴出来跟你在 Word 里手动排版的结果几乎一样。

为什么选客户端路线

做技术分享的人经常要在 LinkedIn 发没公开过的内部数据、客户案例草稿、未发布的 API 设计文档。这些内容每多经过一个第三方服务器，就多一份泄漏风险。md2rich 的卖点不是功能多，是"我的 .md 没离开过浏览器"。页面关闭后所有内容清空，刷新就是空白。

适合什么场景

  • 发技术贴、教程贴、代码讲解，需要代码块和标题层级
  • 写带图表说明的长文（先在 Markdown 里写文字描述，渲染后粘到 LinkedIn）
  • 临时给同事、客户发未公开草稿，敏感内容不想过第三方
  • 多账号运营、团队共享同一份 Markdown 源文件

3. 方案 B：StackEdit（云端，需登录）

StackEdit 是老牌在线 Markdown 编辑器，2015 年就上线了，自带"Publish to LinkedIn"功能，能直接把渲染好的内容推送到你的 LinkedIn 草稿箱。

使用流程：在 app.stackedit.io 注册账号 → 写 Markdown → 点右上角"Publish" → 选 LinkedIn → 授权一次 OAuth → 之后每次写完直接推。

实测问题：

  1. 推送延迟。从点 Publish 到 LinkedIn 草稿箱出现这篇帖子，最快 15 秒，最慢一次等了 3 分钟，期间你盯着屏幕不知道是网慢还是推失败了。
  2. 格式丢失。StackEdit 转 LinkedIn 的代码块会被它自己 wrap 一层 div，LinkedIn 那边偶发渲染成普通段落；引用块样式会塌；多级列表会变成平铺。
  3. 登录依赖。没登录不能 Publish，匿名模式下能写但不能推，2025 年起登录还要求绑定 Google / GitHub 账号。
  4. 服务稳定性。StackEdit 2024 年有过一次长达 6 小时的宕机，期间所有未保存的内容（它不自动 save 到本地）全部丢失。

如果你已经有 StackEdit 账号、对它熟悉、写的都是非敏感内容（公开博客转贴），它仍然是个能用的一站式方案。但要发敏感草稿，或者赶时间发紧急帖子，它就不太合适了。

4. 方案 C：Pandoc + 系统剪贴板（命令行）

Pandoc 是文档转换界的瑞士军刀，理论上它能把 Markdown 转成任何格式。配合 macOS 的 pbcopy 或 Linux 的 xclip，可以做出一个一次性命令：

# macOS
pandoc -s post.md | pbcopy

# Linux（需要 xclip）
pandoc -s post.md | xclip -selection clipboard

# Windows（需要 clip）
pandoc -s post.md | clip

然后到 LinkedIn 编辑器 Cmd+V。看起来很优雅，但实际有几个坑：

  • HTML 直接粘贴不工作。LinkedIn 期望的是语义化富文本，Pandoc 默认输出的是完整 HTML 文档（含 <html>、<body>），粘贴后 LinkedIn 会按纯文本处理。必须用 -s（standalone）参数把 HTML 剪到只剩 body 部分。
  • 代码块样式丢失。Pandoc 转出来的 <pre><code> 没有 LinkedIn 期望的 CSS class，粘贴后变成普通段落。要手动加 --no-highlight 或写 wrapper CSS。
  • 图片必须先托管。Pandoc 处理本地图片是 file:// 引用，LinkedIn 不接受，必须先把图传到 CDN（imgur / S3 / 自家博客），把 URL 替换到 Markdown 里再跑 pandoc。
  • 每次都要敲命令。写完一篇帖子要打开终端、敲一行、看一眼、复制粘贴，比 md2rich 多 5 步。

Pandoc 路线适合有 CI/CD 习惯、已经在跑自动化发布流程的团队工程师，不适合偶尔发贴的内容创作者。

7. 选哪个

如果你只想 5 分钟内把一段 Markdown 发到 LinkedIn、不想注册任何账号、也不想让内容过任何服务器，md2rich 是最快的路径。它没有任何花哨功能，就是把 Markdown 渲染好、复制成富文本、让你贴进 LinkedIn。技术贴、代码讲解、教程长文都能用。

如果你已经在用 StackEdit 写博客、习惯它的界面、能接受内容经过它的服务器，StackEdit 自带的 Publish to LinkedIn 仍然能用，但要对偶尔的格式丢失有心理准备。

如果你是工程师团队、需要批量自动化（比如从 CI 流水线直接发构建结果到 LinkedIn 公司页），Pandoc + 剪贴板路线最灵活，可以塞进任何脚本里。个人偶尔发贴就太重了。

想试 md2rich 的朋友，md2rich.com 直接打开就能用，左边写 Markdown，右边实时预览，点一下按钮复制成富文本，回到 LinkedIn Cmd+V。整个流程不超过 10 秒。
